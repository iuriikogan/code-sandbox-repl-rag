# Agentic RAG Workflow Design

This document details the multi-model architecture and process flow for the Agentic RAG implementation.

## Overview
The Agentic RAG routing process leverages two different Gemini models and localized semantic search to extract high-value information from massive, unstructured datasets at a fraction of the cost of standard processing.

## Components
1. **Orchestrator Agent (`gemini-3.1-flash`)**: The entry point. It receives a prompt, reads the system instructions, and generates a dynamic Python script.
2. **Python Sandbox (`os/exec`)**: An isolated runner executing the script generated by the Orchestrator. 
3. **Embedding Pipeline (`text-embedding-004`)**: Used to convert both the user's query and the chunked context data into vector representations.
4. **Synthesis Agent (`gemini-3.1-pro`)**: The final processing step that reasons over the highly-relevant extracted chunks.

## Process Flow
1. **Initialization**: The Go Host initializes the Orchestrator with the goal, system instructions, and the `execute_python_script` tool.
2. **Script Generation**: The Orchestrator generates a Python script designed to chunk the target dataset and calculates similarity. 
3. **Execution & Embedding (IPC)**:
   - Python chunks the dataset.
   - Python requests a vector embedding for the target query from Go (`{"type": "embed"}`).
   - Python iterates over its chunks, requesting vector embeddings for each from Go.
4. **Semantic Search (Local)**: Python calculates the Cosine Similarity between the query vector and all chunk vectors locally.
5. **Dynamic Selection**: Python drops low-relevance chunks using a dynamic threshold (e.g. > 0.75 similarity score) and returns the top remaining chunks via IPC (`{"type": "done"}`).
6. **Synthesis**: Go intercepts the "done" payload containing the extracted text and sends *only* this highly relevant subset to the `gemini-3.1-pro` model for final analysis and summarization.
